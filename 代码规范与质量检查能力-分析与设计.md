# 代码规范与质量检查能力：分析与设计

> 状态：设计稿（后续开发依据）。
> 目标：在现有全自动交付流水线（plan → deliver → publish → auto-deliver）上，补齐**代码规范识别 + 代码质量检查 + 静态工具执行 + 规范维度独立审查 + 验证记录**，并满足成本控制与"绝不假装"原则。
> 本文基于对当前仓库真实代码的审计（见 §0）。所有路径相对仓库根。

---

## 0. 现状审计（已取证）

| 维度 | 现状 |
|---|---|
| 代码风格/项目规范识别 | **无**。plan 无 `project-code-style` 产出；全仓无 code-style/convention 概念 |
| 静态工具（checkstyle/spotless/pmd/sonar/eslint/prettier/ruff/black/gofmt/golangci 等） | **无**任何检测或执行 |
| 阿里巴巴 Java 规范 | **无**接入，也无预留位 |
| 编译检查 | **无**（Verify 只跑 DONE 测试 + diff，不编译、不静态） |
| deliver 复审视角 | 仅 `correctness / robustness / scope-conformance / risk-coverage`（`deliver-from-plan.js` LENSES_ALL）；无 readability/maintainability/code-style/project-convention |
| 分级 | 复审 `verdict ok|needs-work` + `blocking`；终态 `computeDeliverStatus` 用 blockingReview→BLOCKED、openItems→WITH_OPEN_ITEMS（**可直接复用为 P0/P1/P2 承载**） |

结论：这是**全新增量**，但可**大量复用**现有"独立复审多视角 + blocking 闸门 + openItems 降级 + 独立 Verify 重跑 + 分档(lite/standard/deep)"机制，避免重复造轮子与堆 Agent。

## 1. 规范来源与优先级（清单一）

- **预留接入位、绝不假装已加载**：新增目录 `references/code-style/`，含 `alibaba-java/README.md` 占位说明（明确：规范正文**尚未提供**，此处仅为接入位）。运行时**检测该位置是否有真实规范正文**：
  - 有正文 → 在质量检查/审查 prompt 中作为参考接入，并在产物标注"阿里规范：已接入"。
  - 无正文/仅占位 → 标注"阿里规范：未接入（占位）"，**只用通用 Java 最佳实践兜底，绝不声称已按阿里规范检查**。
- **后续接入方式**：独立规范文件、Skill、或 `references/code-style/` 目录三选一；接入即生效，无需改引擎逻辑（引擎只检测"有无正文"）。
- **优先级（写进 plan/deliver/review 的 prompt 与产物）**：
  ```
  客户项目已有明确规范  >  项目现有代码风格  >  阿里巴巴 Java 规范(若已接入)  >  通用 Java 最佳实践
  ```
- **绝不为符合阿里规范大范围重构客户代码**：编码/修复 prompt 显式禁止；只在 SCOPE 内、最小改动。

## 2. 项目规范识别 → `project-code-style`（清单二）

在 **plan 阶段**新增"项目规范识别"产出 `project-code-style.json`（落 planDir，供方案/编码/审查/修复/验证共用），结构覆盖清单二 12 项：

| 字段 | 含义 |
|---|---|
| `language` / `buildTool` | 主语言、构建工具（maven/gradle/npm/go…） |
| `layout` | 目录与分层结构 |
| `naming` | 类/方法/变量/常量命名约定 |
| `domainObjects` | DTO/VO/BO/Entity 使用方式 |
| `layering` | Controller/Service/Mapper 分层方式 |
| `unifiedResponse` | 统一返回值结构 |
| `exceptionModel` | 异常处理体系 |
| `logging` | 日志格式与级别约定 |
| `transaction` | 事务使用方式 |
| `sqlOrm` | SQL/ORM 使用方式 |
| `commentStyle` | 注释风格 |
| `testStructure` | 测试结构 |
| `staticTools` | 检测到的格式化/静态检查配置与命令（checkstyle/spotless/eslint/...，及 lint/format/verify/check 脚本） |
| `evidence` | 每项判定的出处（文件/配置），不臆测；判不出填 `unknown` |
| `specSource` | 规范来源优先级实际取值（客户规范有无、阿里规范是否接入） |

- 基于**真实读取**目标项目得出；读不到/不确定一律 `unknown`，**不臆测**。
- 非 Java/简单项目：轻量产出（多为 `unknown`/`n.a.`），不强行套 Java 分层术语。
- 这是后续所有阶段判定"是否符合项目风格"的**单一事实源**。

## 3. 编码阶段约束（清单三）

deliver 的 **Implement / 独立 Fix** prompt 增加（**仅改 prompt，不新增 Agent**）：

- 新增代码**优先与 `project-code-style` 一致**（命名/分层/返回值/异常/日志/注释/测试结构）。
- **优先复用**已有工具类/公共组件/异常类/枚举/基础设施（plan.reuse + project-code-style 提供线索）。
- **禁止**引入与现有架构冲突的新分层；**禁止**无需求依据的大规模重构。
- **只改 SCOPE 内文件**（沿用现有 SCOPE 硬闸门）。
- 发现**项目规范 ⟷ 阿里规范冲突**：**遵项目规范**，并把差异记入产物（`styleConflicts`）。

## 4./5./6. 代码质量检查 + 静态工具 + 分级（清单四/五/六）

新增 deliver 阶段 **CodeQuality**（位于 Implement 之后、与 Review 协同；**成本受控、不堆 Agent**）：

**(A) 静态工具执行（确定性优先，单次执行，记录真实命令/退出码/输出）**
- 自动识别并**优先执行项目已有工具/脚本**：Checkstyle / Spotless / PMD / Sonar / ESLint / Prettier / Stylelint / Ruff / Black / gofmt / golangci-lint，以及项目 `lint`/`format`/`verify`/`check` 命令。
- 同时跑**编译**与**测试**（测试沿用 DONE）。
- **静态工具能判的，不再交给多个 AI 重复判**（清单九）。
- **不自行引入新格式化工具、不批量改全项目**；无可用工具→如实记"未执行/无工具"。

**(B) 语义质量检查（工具查不了的，交独立审查，按需而非堆叠）**
覆盖清单四中工具难自动判的项：空指针风险、异常被错误吞掉、日志内容/级别/敏感信息、事务边界、并发/线程安全、**金额计算精度**、SQL 注入/低效 SQL、权限/数据越权、接口字段前后端契约一致、注释与代码一致、命名清晰、结构/分层合理、重复代码、过长方法/职责混乱。
- 这些**折叠进独立 Review 的视角**（见 §7），不另起一支庞大 Agent。

**(C) 分级 P0/P1/P2 → 复用现有终态闸门（不动 core 同源锁）**
每条质量发现带 `severity ∈ {P0,P1,P2}` 与 `blocking`：
| 级别 | 清单要点 | 并入终态 |
|---|---|---|
| **P0** | 编译失败/测试失败/安全漏洞/数据错误或丢失/权限越权/严重事务/严重并发/金额精度错/敏感信息泄露 | `blocking=true` → 经现有 **blockingReview 闸门 → BLOCKED**（绝不交付） |
| **P1** | 异常处理不合理/日志错误/SQL 性能/接口设计/明显空指针/严重重复/分层明显不合理 | **按实际影响**：阻断性→`blocking=true`→BLOCKED；否则→openItems→WITH_OPEN_ITEMS |
| **P2** | 命名/注释/格式/一般可读性/非关键风格不一致 | → openItems（**绝不单独阻断**，且**不得为修 P2 扩大改动**） |

> 复用 `computeDeliverStatus` 现有 `blockingReview`(→BLOCKED) 与 `openItems`(→WITH_OPEN_ITEMS) 通道，**无需改 core 同源块**：P0/阻断 P1 走 blocking review-finding；P1 非阻断/P2 走 openItems。

## 7. 独立审查补充视角（清单七）

在 deliver 现有并行 Review 的 `LENSES_ALL` 增加 4 个视角（**复用现有多视角并行机制，不新增结构**）：
- `readability`（可读性）、`maintainability`（可维护性）、`code-style`（与项目风格一致）、`project-convention`（遵项目既有约定）。

每个视角**必须基于**：项目真实代码 + `project-code-style` + 项目已有**静态检查结果**（§4A 的真实输出）+ 已接入的阿里规范（若 §1 检测到正文）。**禁止只凭模型通用经验下风格结论**（prompt 显式要求引用 project-code-style/静态结果出处）。

**成本分档**（清单九）：`lite` 只保留 correctness+scope-conformance（现状）；`standard/deep` 才加这 4 个 style 视角；**完整规范审查只在复杂或高风险 Java 项目执行**。

## 8. 验证与交付（清单八）

- **独立 Verify 重跑**：编译 + 测试 + 静态检查（与现有"独立复跑 DONE/重物化"同范式，不信实现者自报），记录**真实命令、退出码、输出摘要**；**无法执行的检查标记"未验证"**（不伪造）。
- **写入 `delivery-report.md`**：代码规范结果章节（规范来源、项目规范摘要、检查命令与结果、P0/P1/P2、已修复、剩余、是否影响交付）。
- **`delivery-manifest.json` 新增 `codeStyle` 段**：
  ```
  specSource(规范来源优先级与阿里是否接入) / projectStyleSummary(project-code-style 摘要)
  / checksRun[{command, exitCode, outputTail, status:ran|skipped|unverified}]
  / issues{p0[],p1[],p2[]} / fixed[] / remaining[] / affectsDelivery(bool)
  ```

## 9. 成本控制（清单九）

- **不为代码规范堆重复 Agent**：静态工具单次跑、AI 不复判工具已判项；4 个 style 视角复用现有并行 Review，不另起庞大 Agent。
- **静态工具能判的不交 AI**：AI 只补工具查不了的语义项（事务/金额/越权/契约/注释一致…）。
- **简单任务轻量检查**：`lite` 档跳过 style 视角与重静态；只有**复杂或高风险 Java 项目**才完整规范审查。
- **绝不破坏**现有测试、SCOPE、沙箱、独立验证机制（所有新增都在沙箱内、Verify 之内，SCOPE 不放宽）。

## 11. 诚实边界（与全局纪律一致）

- **规范未接入正文 → 明说"未接入/占位"，绝不声称已按阿里规范检查**。
- **静态工具/编译无法执行 → 标"未验证"**，记录探测依据与缺失，不伪造通过。
- **判不出的项目规范项 → `unknown`**，不臆测。
- 与"无法真实验证须如实写未验证"（项目纪律）、"不静默截断"一致。

## 12. 推荐分阶段计划

- **阶段 S1｜规范接入位 + 优先级（最小、可测）**：建 `references/code-style/alibaba-java/README.md` 占位 + 引擎"检测有无正文"工具 + 优先级常量；self-check 纳入占位文件存在性。
- **阶段 S2｜`project-code-style` 识别（plan）**：plan 产出 `project-code-style.json`（§2 schema）+ 落盘 + 单测（schema/识别样例）。
- **阶段 S3｜静态工具检测+执行（deliver CodeQuality）**：自动识别并跑项目已有工具/编译/测试，记录真实命令/退出码/输出；无工具如实跳过。可在本机用一个含 lint 脚本的玩具项目验证。
- **阶段 S4｜编码约束 + 4 视角审查 + P0/P1/P2 并入终态**：deliver Implement/Fix prompt（§3）；LENSES_ALL 加 4 视角（按档位）；severity→blockingReview/openItems；self-check 绿。
- **阶段 S5｜Verify 重跑静态/编译 + manifest/report 记录**：§8 落地。
- **阶段 S6｜成本分档收尾 + 端到端验证**：lite/standard/deep 裁剪；在玩具 Java/带 lint 的项目上端到端验"P0 阻断 / P2 不阻断 / 无工具如实跳过"。

> 每阶段：改动最小、self-check 保持绿、与并行评审 Batch 文件错开；运行时无法验证的部分如实标注。
