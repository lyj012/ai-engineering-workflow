# 真实浏览器验证能力：分析与设计

> 状态：设计稿（本文为后续开发的依据）。
> 目标：在现有全自动交付流水线（plan → deliver → publish → auto-deliver）基础上，为**前端/全栈 Web 项目**增加一段**真实浏览器验证**——
> 在完成实现、测试、评审、修复、独立代码验证之后、Commit/Push 之前，由**独立验证角色**像真实用户一样打开页面、执行关键交互、检查渲染/控制台/接口/截图，并把结果并入最终交付判定。
> 本文基于对当前仓库真实代码的逐行取证（含对抗式验证）。

理想流程：

```text
用户提交仓库和需求
→ 分析需求和代码 → 设计方案 → 编写测试 → 编写业务代码
→ 独立评审 → 自动修复 → 独立代码验证
→ 真实浏览器验证（本文新增，仅 Web 项目）
→ Commit → Push → 返回完整交付结果
```

---

## 0. 结论先行

现有"独立代码验证"（deliver 的 Verify）是**纯 shell/文件层**：复跑 DONE、独立复现先红后绿、算 diff/SCOPE、从 `test-plan.json` 重物化复测、校验测试指纹。`computeDeliverStatus` 的输入里**没有任何运行时 / HTTP / console / 截图信号**，全仓 `.js/.mjs` 对 `playwright/browser/screenshot/server` **零引用**（仅 `test-planner.md`、全局规则、demo 文案里有"低 token 浏览器流程"的文字建议，无实现）。

要落地浏览器验证，需在**三层**补能力：

- **上游 plan**：判定项目类型（web / 非web）+ 产出"浏览器验收用例"。
- **中游 deliver**：新增独立 `BrowserVerify` 阶段 + 启动项目 + **工具无关**的浏览器驱动 + 独立执行 + 证据落盘。
- **终态判定**：把浏览器结果按**四态**并入交付判定（失败=阻断，无能力=如实跳过降级，非web=不适用，通过=计入绿）。

可复用基座：仓库已有 `openLoopItems → DELIVERED_WITH_OPEN_ITEMS`（只降级不阻断）、独立验证不信自报、"无法验证须如实写未验证"的纪律——但它**只够承载"跳过"一态，不能区分"执行了但失败"**，这是必须新增的闸门。

## 2. 与目标的差距（缺什么）

| # | 缺口 | 层 |
|---|---|---|
| B1 | 无项目类型判定（web/全栈 vs 非web），无处决定"是否要浏览器验证" | 上游 plan |

## 3. 需要新增的核心能力

1. **项目类型识别**（确定性、基于仓库客观结构，不依赖需求文本）：`package.json` 含 react/vue/next/vite/svelte/angular/webpack、`scripts.dev/start/build`、`index.html`、server 入口、端口 → `web | fullstack | non-web | unknown` + 信号清单 + 推断 `startCommand` / `baseUrlGuess`。
2. **浏览器验收用例**（"浏览器版 DONE"）：由需求驱动，描述 打开哪个 URL、做哪些关键交互、期望的 DOM/文本/跳转/刷新后状态、console 应无错、关键接口应 2xx。
3. **项目启动与生命周期**：沙箱副本里 `install + 起 dev/build-serve` → 轮询端口就绪（带超时）→ 拿 baseUrl → 用完清理进程；有界、可观测、失败如实回报。
4. **工具无关浏览器驱动抽象 + 运行时能力探测**（见 §4）。
5. **独立浏览器验证角色**：全新 agent（非实现/修复者），像真实用户跑用例，产出离散结构化结果（见 §6/§7）。
6. **终态四态并入**（见 §6）+ **证据落盘**（截图/网络日志/报告，运行产物 gitignore 不入库）。
7. **诚实跳过专项语义**（见 §8）。

---

## 4. 工具无关的浏览器驱动抽象 + 适配器优先级

定义一个"浏览器验证契约"（输入：baseUrl + 一组浏览器检查；输出：统一结果 schema），由**运行时能力探测**按优先级选适配器，**不绑定单一工具**：

| 优先级 | 适配器 | 运行时探测方式 |
|---|---|---|
| 1 | **Playwright MCP / 任意浏览器类 MCP**（首选，配低 Token 策略：浅快照/仅 console error/过滤网络/终图） | `ToolSearch "browser playwright"`，命中即用 |
| 2 | **Playwright CLI + 可运行浏览器** | `command -v playwright`；`ls ~/.cache/ms-playwright`；必要时 `npx playwright install chromium` 后**实跑一次冒烟确认浏览器真能起** |
| 3 | **Codex 内置浏览器** | `command -v codex` |
| 4 | **系统现代浏览器经 CDP** | 探测 chrome/chromium 版本 |
| 5 | **WebFetch 文本降级** | 仅取渲染后文本，**无 JS/交互/console/network/截图**——只能作"读页面文本"补充，**严禁冒充浏览器验证**，结果须显式标注"仅文本抓取" |
| — | **都没有 → 按 §8 如实跳过** | |

关键纪律：**"真的能起浏览器"才算有能力**——`npx playwright install` 成功 ≠ 浏览器能在本机运行（受老 OS/glibc 约束），必须实跑一次冒烟；否则归"无能力跳过"。

## 5. 项目类型判定放哪

放在**上游 plan 的扫描/preflight 层**（对仓库客观结构识别，与需求无关），由它驱动 test-plan 给出 `verificationType=browser-e2e`，**deliver 侧不重复探测**：

- 把自由文本 `repoKind` 升级为带 enum 的 `projectType`（web/fullstack/non-web/unknown）+ `webSignals[]` + `startCommand` + `baseUrlGuess`。
- `test-planner` 对 web 项目额外产出 `browserChecks[]`（B2）。
- **非 web → 根本不进浏览器验证**（projectType≠web 即 `not-applicable`）。

---

## 6. 独立角色 + 四态并入终态（最关键）

**独立性**：浏览器验证由**全新 agent**完成（非实现/修复者），从需求/plan 的 `browserChecks` 独立驱动浏览器，不采信实现者自检——与现有"独立 Verify 不信自报"同范式（`.claude/workflows/deliver-from-plan.js:405-420`）。可选增强：浏览器版"先红后绿"（新功能 UI 行为在改前版本应缺失/异常、改后应正常），防"改浏览器用例迁就实现"——因改前版本可能起不来，作为加强项而非硬要求。

**四态 → 终态映射**（现 `openLoopItems` 只会降级、不能区分"失败"，必须新增一条阻断闸门）：

| 浏览器验证态 | 触发 | 并入终态 |
|---|---|---|
| `not-applicable`（非 web） | projectType≠web | 无影响 |
| `passed`（已执行且通过） | 启动成功 + 检查全过 + 无 console 错 + 关键接口 2xx | 计入绿 |
| `failed`（**已执行但失败**） | 启动崩 / 检查不过 / console 有错 / 关键接口失败 | **BLOCKED**（新增闸门，类比 `independentTestsPassed===false`；**不能塞 `openLoopItems` 被无声降级**） |
| `skipped-no-capability`（无浏览器能力） | §4 探测全失败 | `openLoopItems` → **DELIVERED_WITH_OPEN_ITEMS**（如实降级，不阻断、不伪造） |

效果：web 项目浏览器验证**真失败 → 不发布**；**无能力 → 带开环项交付并记录剩余风险**；非 web → 不受影响。该闸门在 deliver 终态里，**publish 与 auto-deliver 的现有闸门（要求 DELIVERED/带开环项）自动继承，无需改动它们**。

---

## 7. 需要记录的验证结果（对应交付要求）→ 结果 schema（字段级）

`browserVerify` 结果对象（落 `delivery-manifest` + 独立报告 + 截图目录）：

| 交付要求 | 字段 |
|---|---|
| 是否成功启动项目 | `projectStarted` + `startCommand` + `startLogTail` |
| 是否成功打开页面 | `pageOpened` + `baseUrl` |
| 哪些操作已执行 | `opsExecuted[]`（action / 结果） |
| 哪些验证通过/失败 | `checks[]`（name / passed / detail，覆盖 表单/按钮/弹窗/跳转/刷新后状态/结果是否符合需求） |
| 是否有 Console 错误 | `consoleErrors[]` / `consoleErrorCount` |
| 是否有关键接口失败 | `failedKeyRequests[]`（url / status） |
| 截图与证据位置 | `screenshots[]` + `evidenceDir` |
| 最终浏览器验证状态 | `finalBrowserStatus` ∈ {passed, failed, skipped-no-capability, not-applicable, error} + `adapterUsed` |

证据落盘到运行目录，并按现有纪律 `gitignore`（运行产物不入库）。

## 8. 诚实跳过 / 无法执行（绝不伪造）

web 项目但 §4 探测后无任何可用浏览器能力时，`finalBrowserStatus=skipped-no-capability`，并如实记录：**探测过程与依据**（哪些命令/查询返回空）、**缺失的具体能力**、**剩余风险**（UI/JS/交互未经真实浏览器验证）、**补全建议**（联网 `npx playwright install chromium` 后重试，或换有浏览器 MCP 的环境复验）。**严禁**用 WebFetch 文本冒充"已通过浏览器验证"；WebFetch 若用，必须标注"仅文本抓取，非浏览器验证"。对应顶层纪律（"无法真实验证时必须如实写未验证及原因"）。

> 注：当前开发环境恰好"无任何可用真实浏览器能力"（无浏览器 MCP、未装 Playwright/Chromium、仅有过旧 Firefox 88、无 Codex CLI），正是这条分支的真实样例——验证实现时应确保此场景判 `skipped-no-capability` 而非伪造通过。

## 9. 插入点与架构（推荐方案 + 两条硬约束）

**推荐：在 `deliver-from-plan` 新增独立 `BrowserVerify` 阶段**，位于独立代码 Verify 之后、Deliver 落盘/终态计算之前（`.claude/workflows/deliver-from-plan.js` 的 Verify 段之后），由独立 agent 在沙箱副本上执行。理由：deliver 手上就有"可运行的实现代码"（沙箱），把浏览器结果并入 deliver 终态后，publish/auto-deliver 的发布闸门**零改动**自动生效。

- **备选**：独立引擎 `browser-verify.js`，由 auto-deliver 在 Deliver 与 Publish 之间调起。独立性更强，但要重新把项目跑起来（重复 clone+apply），并改 auto-deliver/publish。**首版不建议**，复杂度不成比例。

**两条必须遵守的硬约束**：

1. **同源锁**：`computeDeliverStatus` 被 `scripts/self-check.mjs` 锁定（`core/deliver-status.mjs` ⟷ deliver 内联副本 ⟷ 单测三处一致）。加 `browser` 输入与 `failed→BLOCKED` 闸门时，**必须三处同步改 + 补单测**，否则 self-check 红。
2. **确定性**：浏览器验证天然非确定，而 Workflow 脚本体禁 `Date.now/Math.random`、要可断点恢复。做法：**非确定的浏览器操作全在 agent 内完成，只把离散结果（布尔/计数/路径）喂给纯函数终态判定**；时间戳/截图命名用 agent 的 Bash `date`，不进脚本体。决策层因此仍确定、可缓存复跑。

## 10. 上游 plan 侧配套改动

- `plan-from-requirement`：scan/preflight 产出结构化 `projectType` + `webSignals` + `startCommand` + `baseUrlGuess`；`test-planner` 对 web 项目产出 `browserChecks[]`（把需求关键交互翻成可执行的浏览器验收）。
- 这是 B1/B2 的落点，让 deliver 的 BrowserVerify "有据可依、不临时拍脑袋"。

## 11. 推荐分阶段计划

- **阶段 A｜判类型 + 出用例（上游 plan）**：`projectType` enum + `webSignals` + `startCommand`；test-planner 产 `browserChecks`。低风险，先固化"是不是 web、要验什么"。
- **阶段 B｜能力探测 + 工具无关驱动契约**：实现 §4 运行时探测与适配器选择 + 统一结果 schema + 诚实跳过语义。可先做 `core/browser-capability` 探测 + `browser-verify` 子代理 prompt 契约，在"无能力→跳过"和"有 MCP→真验"两端各跑一次。
- **阶段 C｜deliver 接 BrowserVerify 阶段 + 终态四态**：插阶段；三处同步改 `computeDeliverStatus`（加 `browser` 输入 + `failed→BLOCKED` + `skip→openItem`）+ 补单测；扩 manifest/openItems；self-check 保持绿。
- **阶段 D｜真机端到端验证**：造一个**最小 web 玩具仓库**（如 Vite/React：一个按钮 + 一次 fetch）放进 `examples/`，跑 `auto-deliver`：在有浏览器能力的环境验"真验通过 / 真失败阻断"，在无能力环境验"无能力如实跳过"。这是唯一能暴露运行时问题的验证。
- **阶段 E（可选）**：publish 后对部署 URL 的浏览器冒烟（需要 staging/prod URL，当前流水线不管理部署，**默认不做**）。

---

## 12. 关键风险与约束

- **非确定性 × 确定性体系**：必须把浏览器观测降为离散信号再喂纯函数（§9 约束 2），否则破坏断点恢复与同源锁。
- **能力探测的"假阳性"**：`npx playwright install` 成功 ≠ 浏览器能跑，必须实跑冒烟才算有能力，否则归"跳过"。
- **成本**：起 dev server + 装浏览器 + 跑 e2e 比 shell DONE 重得多；需有界超时 + 低 Token 策略（浅快照 / 仅 console error / 过滤网络 / 终图）。
- **"失败 vs 跳过"绝不可混**：诚实性核心——失败必须阻断、跳过只能降级，二者用不同 `finalBrowserStatus` 与不同终态分支承载。
