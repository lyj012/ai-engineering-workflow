# Workflow 设计清单（设计编排时逐项过）

> 配合 `SKILL.md` 的设计主干使用。每一项都要能回答“是/否/不适用 + 依据”。

## A. 任务定义（动手编排前）
- [ ] 目标一句话能说清；范围、非目标明确。
- [ ] 最终交付物可枚举、可验收。
- [ ] 评价标准可观察（不是“做得好”这种）。
- [ ] 主要风险与待确认问题列出。
- [ ] 判断过：这真的需要 Workflow 吗？单 agent 能干完就别上。

## B. 阶段拆分（每个 phase 都要过）
- [ ] **输入**：这一阶段消费什么（上一阶段产物 / args / 文件）。
- [ ] **输出**：产出什么（最好是 schema 结构化对象）。
- [ ] **完成标准**：怎样算这一阶段做完。
- [ ] `meta.phases[].title` 与脚本 `phase('...')` 逐字一致。
- [ ] 阶段之间是真有依赖（需串行/栅栏），还是可并行/流水。

## C. 控制流
- [ ] 默认 `pipeline()`；用了 `parallel()` 栅栏的，能说出“为什么必须全量一起”。
- [ ] 有循环的：写明**最大轮次**与**退出条件**（loop-until-dry / count / budget），杜绝无限循环。
- [ ] 纯 transform（map/filter/flatten）放进 stage，而不是用栅栏断开。
- [ ] 截断/采样（top-N、抽样）有 `log()` 显式说明被丢弃的覆盖面。

## D. Subagent 分工
- [ ] 每个 agent 一个清晰、单一、可命名的职责。
- [ ] 无职责重复、无关键环节遗漏。
- [ ] 用 `schema` 返回结构化结果（除非确实只需自由文本）。
- [ ] `label` 可读，便于 `/workflows` 追踪。
- [ ] 选了合适的 `agentType`（只读广搜→Explore；通用→general-purpose；Claude Code 问答→claude-code-guide；自定义角色→项目 agents）。

## E. 质量门禁
- [ ] **实现者不自评**：有独立 agent / 独立 Workflow 阶段做审查。
- [ ] 高风险结论用对抗式（多 skeptic 证伪）或多视角（不同 lens）验证。
- [ ] 评审-返工是**有界循环**（最多 N 轮），超限如实报告未完成，不谎称完成。
- [ ] 任何 P0 → FAIL 的判级规则明确。

## F. 状态、上下文与恢复
- [ ] 关键中间结果落 `evidence/`，可追踪、可复盘。
- [ ] 上下文收敛：子代理隔离 + schema，避免把整文件塞回主流程。
- [ ] 需要时支持 `resumeFromRunId` 断点恢复（脚本走 `scriptPath`）。

## G. 成本与规模
- [ ] agent 数量与任务规模匹配，**没有为展示而堆 agent**。
- [ ] 分层选 `model`/`effort`：便宜机械活用低配，难的验证/评审用高配。
- [ ] 大规模或预算敏感时用 `budget` 设硬上限（先判 `budget.total` 非空）。
- [ ] 估算过：最坏情况 agent 总数 / Token 量级是否可接受。

## H. 验证与交付
- [ ] 真实跑过一遍，留运行记录（不是“应该能跑”）。
- [ ] 无法验证的部分如实标注“未验证 + 原因”。
- [ ] 交付物路径自洽、运行命令可照抄、他人可复用。
