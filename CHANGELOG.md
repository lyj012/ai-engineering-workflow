# Changelog

## Unreleased

- Surface the browser verification result in `auto-deliver`'s final returned result and report: the end-to-end summary's `delivery.browserVerify` now carries `applicable` + `finalBrowserStatus` (passed/failed/skipped-no-capability/not-applicable/error) + adapter + evidence dir + check counts + console-error count + failed key requests + screenshots, and the auto-deliver report explicitly states the final browser-verification status (non-web → not-applicable; no capability → skipped with residual risk). Completes the "return a complete delivery result including the final browser-verification status" requirement at the orchestrator level.

- Refine the BrowserVerify prompt for partial-capability browser adapters: a render/screenshot-only browser (e.g. `firefox --headless --screenshot`) can verify page load/render/console but cannot drive scripted interactions, so required interactions it cannot exercise are now recorded as honest open items (`finalBrowserStatus` = `error` → DELIVERED_WITH_OPEN_ITEMS), never faked as `passed`; a broken/blank render still yields `failed`. Validated at runtime on a web toy with no MCP/Playwright/Codex available: the verifier genuinely fell back to Firefox headless, started the live dev server, rendered the page, and correctly flagged a non-rendering page as `failed` — no faking, fully tool-agnostic.

- Add the `BrowserVerify` stage to `deliver-from-plan` (web/fullstack targets only). After the independent code Verify, an independent agent: detects project type (the deterministic `classifyProjectType` is now also inlined under PROJECT-TYPE markers and parity-locked against `core/project-type.mjs`); for non-web targets records `not-applicable` and skips; for web targets probes browser capability in a tool-agnostic priority order (Playwright MCP → Playwright CLI → Codex built-in → other), and if none is usable records `skipped-no-capability` honestly (never faking success, never substituting WebFetch text); when a browser is available it starts the project, opens the page, performs the requirement's key interactions like a real user, and records project-started, page-opened, executed ops, per-check pass/fail, console errors, failed key requests, screenshots, and a final status. The four-state result feeds the deliver gate: `failed` → BLOCKED (a broken web delivery is never published), `skipped`/`error` → DELIVERED_WITH_OPEN_ITEMS, `not-applicable`/absent → no effect. All of it is recorded in the delivery manifest. Self-check stays green (syntax + project-type parity); the runtime passed/failed paths require a browser-capable environment to exercise end to end (this repo's current env exercises only the honest-skip path).

- Wire the real-browser verification result into the deliver final-status decision: `computeDeliverStatus` now accepts an optional `browser` input (web projects only). `applicable + failed` → BLOCKED (a broken page/interaction/console/API never gets published); `skipped`/`error` or carried open items → DELIVERED_WITH_OPEN_ITEMS (honest skip — recorded, not faked, not silently blocked); `not-applicable` or absent → no effect (backward compatible). Kept in sync across `core/deliver-status.mjs`, the inline copy in `deliver-from-plan.js`, and unit tests; self-check parity green. This is the deterministic gate the upcoming BrowserVerify stage feeds.
- Begin real-browser verification (see `真实浏览器验证能力-分析与设计.md`): add `core/project-type.mjs`, a deterministic classifier that decides from objective repo signals (package.json deps/scripts, index.html, server entry) whether a target is a browser-verifiable `web`/`fullstack` app or `non-web`/`unknown`, and derives a `startCommand`/`baseUrlGuess` for web targets. This scopes which deliveries should run browser verification (web only) vs skip it (non-web). Unit-tested and covered by self-check.
- Strengthen the `git-guard` PreToolUse red-line classifier: evaluate each shell segment independently so an unrelated flag after `&&`/`;`/`|` (e.g. a trailing `rm -f`) is no longer a false positive, while a forbidden git op anywhere in a compound command stays blocked; also block history rewrites (`git filter-branch` / `filter-repo`) and force branch deletion (`git branch -D`). Ignore `evidence/publishes/` and `evidence/auto/` run outputs.
- Add `auto-deliver` end-to-end orchestrator: from a single requirement plus a target repository it chains `plan-from-requirement` → readiness gate → `deliver-from-plan` → delivery gate → `publish-delivery` using one level of `workflow()` (the three engines never call `workflow()` themselves, so the one-level-nesting limit is respected). Gates are deterministic; the run pauses only on a NEEDS_CLARIFICATION requirement, a plan red line, or missing push permission, and otherwise advances automatically. Children resolve by name (when the repo is the active project) with a `scriptPath` fallback via the optional `workflowsDir` arg so the orchestrator also runs from outside the repo's project context. Validated end to end against a local throwaway repo (requirement → branch/commit/push to a local remote, independently verified).
- Add `publish-delivery` workflow: turn a verified delivery (changes.diff + `DELIVERED`/`DELIVERED_WITH_OPEN_ITEMS` manifest) into an automatic git branch/commit/push, then verify the remote independently (branch SHA matches the commit, committed files match the delivery, no forbidden files, clean tree). Clones the remote into an isolated working copy (the original target repo is never touched), never force-pushes, refuses to push `main`/`master`/`release` unless `gitPolicy.allowMainPush`, keeps high-risk domains (payment/permission/secret/auth) behind a human gate by default, and never embeds credentials (push relies on the ambient SSH/credential helper). Deterministic status decision lives in `core/publish-status.mjs` with unit tests and self-check parity; no PR is created.
- Harden `deliver-from-plan` role separation and test integrity: the fixer is now an explicit independent role (minimal-scope, must not touch tests); the implementer and fixer are forbidden from editing the materialized tests; independent Verify re-materializes the new-feature checks from `test-plan.json` and re-runs them against the sandbox, blocking delivery when they fail (prevents weakening tests to make DONE pass); a frozen tests fingerprint flags in-tree test changes; review-completeness is now tracked cumulatively across rounds, not only the last.
- Remove default dependencies on author-local parent or sibling directories.
- Add public quick start, examples, sanitized artifacts, and repository self-checks.
- Add open-source project files for contribution and security reporting.
