# 01 Workflow 总览与构件选型

> 本文是整套方法论的入口。读完应能回答两个问题：**Dynamic Workflow 到底解决什么问题**，以及面对一个任务时**该用 Skill / Subagent / Workflow / Hooks / CLAUDE.md 中的哪一个**。
> 凡涉及 Workflow 脚本 API 的细节，一律以 `evidence/01-workflow-api-ground-truth.md`（产品内一手规范）为准；本文只做概念与选型，不复述 API 细则。

---

## 1. 什么是 Dynamic Workflow，它解决什么问题

### 1.1 一句话定义

Dynamic Workflow 是一段**纯 JavaScript 编排脚本**：它决定"下一步派哪个子代理、跑什么、按什么顺序、何时汇总"。决定下一步做什么的是**脚本（确定性代码）**，不是模型的临场判断。脚本会被持久化、可断点恢复，能协调数十到上百个子代理。

### 1.2 它针对的痛点

单个 agent 在一个对话里硬扛复杂任务时，会撞上四堵墙：

1. **上下文装不下**：一次迁移 / 审计 / 广扫要看的文件、要核的结论，远超单个上下文窗口；塞进去就互相挤压、互相污染。
2. **顺序不可控**：靠模型"自己记得先做 A 再做 B"，长流程里必然漂移、漏步、重复。
3. **不可复现 / 不可恢复**：跑到一半中断或想改第 5 步，只能从头再来，前面的成本全部作废。
4. **质量靠自觉**：让同一个 agent 既实现又自评，等于让它给自己打分，发现不了自己的盲区。

Dynamic Workflow 用四个手段对症下药：

- **隔离**：每个子代理有独立上下文窗口，干完只回传压缩后的结果，而不是把一堆中间过程堆回主线程。
- **确定性编排**：串行 / 并行 / 分支 / 循环 / 汇总 / 重试由脚本写死，不靠模型临场记忆。
- **持久化 + 断点恢复**：脚本落盘，未改动的 `agent()` 前缀在恢复时瞬时命中缓存，只重跑改动及之后的部分。
- **独立质量门禁**：实现者与评审者是不同实例，可叠加对抗式证伪、多视角验证、有界返工循环。

### 1.3 什么时候**不该**上 Workflow

- 一个 agent 单步就能干完 → 直接做，别上 Workflow。
- 一行小改、单纯解释代码、回答概念问题 → 上 Workflow 是过度设计。

判断口径很简单：**需要"分解 + 覆盖"、需要多个独立视角、或单个上下文明显装不下**，才用 Workflow。建议先"贴边侦察"（列文件、定范围）拿到工作清单，**再**编排——你不必在动手前就知道任务的全部形状，只需在**编排那一步之前**知道工作清单。（设计主干详见 docs/02、docs/04。）

## 2. 五构件对照表

Claude Code 把"复杂工程能力"拆成五个互补构件。混用会出问题：把固定约束写进 Skill 它不强制，把确定性门禁写进 CLAUDE.md 它会被对话冲淡。先看职责边界：

| 构件 | 一句话定义 | 谁持决策权 | 加载 / 执行时机 | 是否强制 | 典型适用 | 反例（别这么用） |
|---|---|---|---|---|---|---|
| **Skill** | 按需加载的方法 / 规范 / 清单（知识库） | 模型（决定何时调用） | 会话启动只见**描述**；被调用时才载入全文 | 否（顾问性上下文） | 设计清单、编码规范、评审 rubric、领域 SOP | 当作硬门禁——它不强制，模型可以不调用 |
| **Subagent** | 独立上下文 + 自定义系统提示 + 受限工具的专业执行者 | 模型 / 父 Workflow（决定派不派、派谁） | 被 `agent()` 或主代理派生时启动；完成回传压缩摘要 | 否 | 研究、广搜、分析、独立审查、隔离的写文件操作 | 让同一个实现 subagent 给自己的产出打分（实现者自评） |
| **Workflow** | 纯 JS 编排脚本（阶段 / 串 / 并 / 分支 / 循环 / 汇总 / 重试） | **脚本本身**（确定性代码，非模型临场判断） | `Workflow` 工具调用后**立即返回 runId 并在后台运行**，完成时通知 | 编排顺序强制；内容质量仍靠子代理 | 迁移、审计、广扫、长流程、需并行 / 分支 / 循环 / 独立评审 | 一个 agent 单步能完成却硬上 Workflow；为展示规模堆 agent |
| **Hooks** | 生命周期特定点的确定性自动化，由 **harness** 执行 | **harness**（不是模型） | 在生命周期挂载点（如 PreToolUse）自动触发；`exit 2` 可阻断 | **是**（强制层） | 提交前扫描、强制格式化、技术上拦死某类操作 | 用它承载"建议性偏好"——它是硬拦截，不是软提示 |
| **CLAUDE.md** | 项目级固定约束（每次启动加载的上下文） | 模型（读取后自觉遵守） | 每次会话启动加载；跨 `/clear` 与重启存活 | 否（是上下文，非技术强制） | 不随对话变化的项目硬规则、边界、提交纪律 | 把"必须无条件生效的红线"只写在这里——它可被对话稀释，红线要靠 Hook |

> **关键区分**：Skill 和 CLAUDE.md 都是**顾问性上下文**——模型读到、通常会遵守，但**不是技术强制**。要让一条红线无条件生效，必须用 PreToolUse Hook（`exit 2`）或 `permissions.deny` 在 harness 层技术强制；CLAUDE.md 同步写明意图即可，但不能只靠它。（依据见 `evidence/03-decision-log.md` 采用结论 8、9。）

## 3. 选型决策树与口诀

### 3.1 口诀

> **知识找 Skill，干活找 Subagent，编排找 Workflow，必然发生找 Hooks，固定约束找 CLAUDE.md。**

### 3.2 决策树

从你手里的"东西是什么"出发，顺着问下去：

```
你要落地的是什么？
│
├─ 是一段"必须每次都确定性发生"的检查/自动化（提交前扫描、格式化、拦死某操作）？
│     └─ 是 → Hooks（harness 执行，exit 2 阻断）—— 这是唯一的强制层
│
├─ 是一条"不随对话变化、每次启动都要遵守"的项目固定约束？
│     ├─ 需要技术上无条件拦死 → 仍然是 Hooks / permissions.deny
│     └─ 是顾问性硬规则、靠模型自觉 → CLAUDE.md（同步在 Hook 兜底红线）
│
├─ 是一份"方法 / 规范 / 清单 / 领域知识"，希望模型按需取用？
│     └─ 是 → Skill（启动只见描述，调用才载全文）
│
└─ 是一个"要执行的任务"？
      │
      ├─ 一个 agent 单步就能干完？
      │     └─ 是 → 直接做（别上 Workflow）；若需独立上下文/受限工具 → 单个 Subagent
      │
      └─ 需要分解+覆盖 / 多个独立视角 / 单上下文装不下 / 要并行·分支·循环·独立评审？
            └─ 是 → Workflow 编排（脚本用 agent()/pipeline()/parallel() 协调多个 Subagent）
```

### 3.3 一个混用的反例（说明边界为什么重要）

把"提交前必须通过 presubmit 扫描"这条红线**只**写进 CLAUDE.md，是无效的——它是顾问性上下文，模型可能在长对话中忽略它。正确做法：扫描脚本由带 Bash 工具的 Subagent 调用回报，而**真正的强制门禁落到 PreToolUse Hook**；CLAUDE.md 里同步写明这条意图。注意：Hook 的**具体配置必须先读项目实际 `settings.json`**，不能照抄。（依据见 `evidence/03-decision-log.md` 采用结论 10、未确认项 4。）

本套文档按"概念 → 设计 → 实现 → 质量 → 案例"递进。各篇分工如下：

| 文档 | 主题（与文件 H1 一致） | 讲什么 |
|---|---|---|
| **01**（本文） | 总览与构件选型 | Dynamic Workflow 是什么、解决什么问题；五构件对照与选型决策树 |
| **02** | Workflow 阶段拆分与控制流技巧 | `pipeline`（默认）/ `parallel`（栅栏）/ 分支 / 循环（loop-until-dry/count/budget）的选择依据、墙钟特性与可照抄骨架 |
| **03** | Workflow 设计规范与硬约束 | 脚本硬约束、编排原语速查、并发/容量/预算上限、每阶段输入/输出/完成标准模板 |
| **04** | Subagent 协作与分工 | 一个 agent 一个清晰职责、无重复无遗漏；内置 vs 自定义 `agentType` 的可移植性取舍；实现者与审查者分离 |
| **05** | 状态、上下文与成本控制 | 中间结果落 `evidence/`；`{scriptPath, resumeFromRunId}` 断点恢复；并发上限、`budget` 硬上限、模型 / effort 分层 |
| **06** | 验证、交叉审查、失败重试与退出条件 | 实现者不自评、对抗式证伪、多视角验证、评审-返工有界循环；P0 判级 |
| **07** | 反模式 | 不必要的栅栏、为展示堆 agent、实现者自评、静默截断、无退出循环等真实案例 |
| **08** | 质量评价与排名评估标准 | 100 分制六维权重、P0→FAIL 判级、需求覆盖矩阵、用于团队排名评估 |
| **09** | 复用现有 ai-engineering-delivery-zh Skill | 把 8 阶段交付思想映射成 Workflow 的 phase（8→4 折叠），不照搬业务、不改原 Skill |
| **10** | 最小可运行示例：analyze-repo | 最小示例的 7 阶段设计 + **真实运行结果**、运行方式、与方法论的对应自检 |
| **11** | 主工作流：plan-from-requirement | 需求→可执行方案（澄清闸门/快路径/完整流程三级分级），与 analyze-repo 共享引擎 |
| **12** | 桥接：plan → 编码到测试全绿 | 把方案接到"写码闭环"（沙箱内、出 diff、不提交）；就绪闸门、测试物化、复用 `vendor/zhuliming-templates` |
