# 08 质量评价与排名评估标准

> 本文是 Workflow 方法论交付物的**深度评价基线**，也是团队**排名评估**口径。
> 精简版（最终评审用的速查表）见 `.claude/skills/workflow-designer/references/review-rubric.md`；
> 执行评审的角色定义见 `.claude/agents/workflow-reviewer.md`（关系详见本文末「与 workflow-reviewer agent 的关系」）。
> 凡涉及 Workflow 脚本 API 的判断，一律以 `evidence/01-workflow-api-ground-truth.md`（产品内一手规范）为准。

## 一、为什么需要一份显式 rubric

Workflow 交付物（文档 / Skill / Subagent / Workflow 脚本 / 运行记录 / 证据）很容易在两个方向上失真：

- **看起来很多**：agent 数量多、代码行数多、运行时长久——但其中可能是「为展示规模堆 agent」的反模式（见 `docs/07`）。
- **看起来很对**：方案描述漂亮、自称「已验证 / 测试通过」——但没有真实运行记录与产物。

本 rubric 的目标是把评价从「印象」拉回到「逐项需求 + 真实证据」，并且**可量化、可复现、可用于横向排名**。核心纪律来自 `evidence/03-decision-log.md` 跨切原则第 6 条：**质量靠独立性而非数量**；以及 `CLAUDE.md` 三-2：**不以运行时长或代码量充当质量**。

## 二、评分维度与权重（100 分制）

与 `review-rubric.md` 和 `workflow-reviewer.md` 完全一致，不另立口径：

| 维度 | 分值 | 评分焦点 |
|---|---:|---|
| 需求覆盖与正确性 | 30 | 逐项对照原始需求；用覆盖矩阵呈现 met/partial/missing |
| 可运行性与真实证据 | 25 | 有真实运行记录与产物，而非「应该能跑」 |
| 阶段 / 职责设计 | 15 | 每阶段输入 / 输出 / 完成标准；无重复无遗漏 |
| 健壮性 | 10 | 状态记录、失败重试、退出条件、无死循环、降级 |
| 复用现有 Skill | 10 | 复用而非复制，且未破坏现有资产 |
| 成本与规模克制 | 10 | 无为展示堆 agent；上下文 / Token 可控 |

> 权重设计意图：需求覆盖（30）+ 真实证据（25）合计 55 分，让「真的做到了 + 真的验证过」占主导，压制「设计华丽但空转」的交付。

### 各维度内部建议刻度（用于打分一致性）

每个维度内部按「**达标证据**」给满分区间，按「**反模式**」扣分。建议四档：满分（证据齐全且优秀）/ 大部分（小瑕疵）/ 半数（partial 较多或证据偏弱）/ 不及格（缺失或踩反模式）。**任何 P0 直接触发判级规则的 FAIL，与分数无关**（见第三节）。

## 三、判级规则（硬性，与精简版一致）

判级**优先于分数**：

- **任何 P0 → FAIL**（无论总分多高）。
- 无 P0，但存在阻断性 P1 / 关键证据缺失 → `CONDITIONAL_PASS`。
- 无 P0、无阻断性 P1、证据齐全、分数达标 → `PASS`。

严重度定义（与 `workflow-reviewer.md` 一致）：

- **P0**：需求未满足 / 不可运行 / 谎称已验证 / 有死循环 / 复用造成破坏。
- **P1**：影响质量或复用，但有可行修复。
- **P2**：改进项。

> 为什么 P0 一票否决而不看分数：一个「谎称已验证」或「会死循环」的交付，即使其他维度满分，对负责人也是负资产。分数用于**同为 PASS / CONDITIONAL_PASS 的交付之间排序**，不用于把 FAIL 救回来。
>
> 「分数达标」的阈值（如总分 ≥ 多少判 PASS）属团队约定，本文不写死具体门槛数字——这类阈值若无团队共识即写死，会变成虚假精度（参照 `decision-log` 否决项对自创量化公式的态度）。建议团队首次评审时先抽样校准，再固定阈值。

## 四、需求覆盖矩阵模板

需求覆盖矩阵是 30 分维度的核心产出，也是排名时最硬的可比项。每条需求一行：

| 需求项（原文摘要） | 状态 | 证据（文件 / 产物 / 行号） | 备注 |
|---|---|---|---|
| 需求 1：…… | met | `docs/NN.md` / `evidence/XX.md` 第 N 行 | |
| 需求 2：…… | partial | `…` | 缺 XX，建议补 |
| 需求 3：…… | missing | — | 未实现 → 至少 P1，影响核心则 P0 |

填写纪律：

1. **状态只取 met / partial / missing 三值**，不允许「基本完成」之类模糊词。
2. **每条 met / partial 必须指向具体文件 / 产物 / 行号**；指不出证据的，按 missing 或 partial 处理，不得采信自述。
3. **核心需求 missing → P0**；非核心 missing 或多处 partial → P1。
4. 矩阵的「需求项」一列应**逐字来自原始需求 / 验收清单**，不得由评审者改写后再自评。

## 五、每个维度的「达标证据 vs 反模式」

> 「达标证据」是给满分需要在交付物里**实际指得出**的东西；「反模式」命中即扣分，标 ★ 的命中即构成 P0。

### 1. 需求覆盖与正确性（30）
- **达标证据**：完整覆盖矩阵；每条 met 有可点开的产物；正确性结论与 `01-workflow-api-ground-truth.md` 逐字一致（如 `.filter(Boolean)` 只用于 `pipeline()/parallel()` 整体返回数组、不进 stage 内部；`meta` 纯字面量；phase title 与 `phase()` 逐字一致；禁 `Date.now()/Math.random()/无参 new Date()`；脚本体不读写文件）。
- **反模式**：★核心需求 missing；★把与 ground-truth 冲突的 API 写法当作正确（如在 stage 内对标量 `prevResult` 调 `.filter`）；用「应该覆盖了」代替逐项核对；把网络旧文档凌驾于一手锚点。

### 2. 可运行性与真实证据（25）
- **达标证据**：`evidence/` 下有**真实运行记录**——runId / scriptPath、返回结果、`log()` 叙述、产物文件、退出码或截图；未验证之处**如实写明「未验证」及原因**（符合 `CLAUDE.md` 三-4 与 `decision-log` 原则）。
- **反模式**：★在无运行记录的情况下声称「已验证 / 测试通过」（直接 P0，谎称已验证）；只贴脚本不贴运行产物；把「代码看起来对」当作「跑过了」。
- **排名提醒**：本维度只认产物，不认篇幅。没有 evidence 产物的交付，本项按未验证给低分，无论描述多详尽。

### 3. 阶段 / 职责设计（15）
- **达标证据**：每个 phase / 每个子代理有明确**输入 / 输出 / 完成标准**；阶段间无职责重复、无遗漏；并发选择有依据（默认 `pipeline()`，只有需要全量跨 item 结果时才用 `parallel()` 栅栏——与「谁更快」无关）。
- **反模式**：阶段职责含糊、互相重叠或留空白；不必要的 `parallel()` 栅栏；把纯 transform 用栅栏断开而非内联进 stage；让实现者自评自己的阶段产物。

### 4. 健壮性（10）
- **达标证据**：有状态记录与断点恢复设计（`{scriptPath, resumeFromRunId}`）；失败分支有处理（如 `agent()` 返回 `null` 时短路或 `.filter(Boolean)` 整体过滤）；循环有**最大轮次 + 可观测退出**（如最大 3 轮、某轮新增=0 即停，并标注「示例缺省值，非 API 约束」）；超 budget 有降级（主动查 `remaining()`）。
- **反模式**：★无最大轮次 / 无退出判据的循环（死循环风险，P0）；忽略 `null` 返回导致后续 NPE；budget 硬上限触发后无降级路径。

### 5. 复用现有 Skill（10）
- **达标证据**：复用既有 Skill / Subagent / Util 的「按需加载、单一事实源、任务分级、规则优先级链」等能力，而非重写；明确**未修改 / 未破坏**受保护资产（`CLAUDE.md` 一-1 列出的 Skill）。
- **反模式**：★改动或覆盖受保护 Skill 造成破坏（P0）；复制粘贴既有逻辑另起一份；该用 Hook 强制的红线却只写进 CLAUDE.md（CLAUDE.md / Skill 是顾问性上下文、非强制）。

### 6. 成本与规模克制（10）
- **达标证据**：agent 数量按「**是否新增独立视角**」分配，每个 agent 有可说明的独立职责；fan-out 前设 budget；effort / 模型分层；worktree 默认不用；被丢弃的覆盖面有 `log()`（不静默截断）。
- **反模式**：为展示规模堆同质 agent（如研究类 fan-out 按「主题数量」而非「独立视角」切分，参见 `02-research-findings` 自陈的同质性教训）；静默截断 top-N / 采样；以运行时长 / agent 数 / 代码行数自证质量。

## 六、如何用于团队排名评估

排名评估的目标是在**多份交付物之间**给出可解释、可复现的高下，而不是给单份打个模糊印象分。

### 6.1 排名流程
1. **同一份原始需求**下，对每份交付物各跑一次独立评审，产出：verdict + 分项小计 + 总分 + 覆盖矩阵 + P0/P1/P2 清单。
2. **先按 verdict 分层**：FAIL < CONDITIONAL_PASS < PASS。低层不因总分高而超过高层（一个 FAIL 不能因为分高排到 PASS 之前）。
3. **同层内按总分排序**；总分相同再比「需求覆盖与正确性 + 真实证据」两项小计（合计 55 分，代表「真做到 + 真验证」）。
4. **每个名次都要附证据**：排名结论必须能指向覆盖矩阵与 evidence 产物，禁止只给名次不给依据。

### 6.2 防「只追时长 / 代码量」的硬约束
- **时长、agent 数、代码行数、文档字数一律不进评分**，仅作为「成本与规模克制」维度的**反向**观察点（异常膨胀反而扣分）。
- 评分只认三类硬证据：**覆盖矩阵的 met 比例**、**evidence 里的真实运行产物**、**与 ground-truth 一致的正确性**。
- **同质堆叠惩罚**：多个 agent / 多份文档若内容高度重复（无新增独立视角），按「成本与规模克制」反模式扣分，并在排名备注中标出（参照 `02-research-findings` 的同质性自陈案例）。
- **未验证不得冒充已验证**：任何「已验证 / 测试通过」若无 evidence 产物支撑，触发 P0 → 该交付物直接落 FAIL 层，排在所有 PASS / CONDITIONAL_PASS 之后。

### 6.3 可量化、可复现
- 同一份交付物由不同评审实例评，verdict 与分项应可复现（差异只应来自证据本身，不来自评审者偏好）。
- 排名表建议字段：`交付物` / `verdict` / `总分` / `需求覆盖小计` / `真实证据小计` / `P0数` / `P1数` / `关键证据指向` / `一句话结论`。
- 阈值（PASS 总分线、同质重复的判定）由团队**事先约定并固定**，评审中不临时改尺，避免可比性被破坏。

## 七、与 workflow-reviewer agent 的关系

| 角色 | 文件 | 定位 |
|---|---|---|
| 评分标准（本文） | `docs/08-evaluation-rubric.md` | 深度版评价 / 排名口径，给人读、给团队对齐 |
| 速查表 | `.claude/skills/workflow-designer/references/review-rubric.md` | 评审时的精简执行清单 |
| 执行评审的人 | `.claude/agents/workflow-reviewer.md` | 独立子代理，按 12 项必查实际打开产物核查 |

- **三者口径一致**：维度、权重、判级规则、严重度定义在三处完全相同；本文是「为什么这样评 + 如何用于排名」的展开，不引入新的相互矛盾的标准。
- **实现者不得自评**：本 rubric 由独立的 `workflow-reviewer`（与实现者**不同实例**）执行；实现者不得拿本文给自己打分（`CLAUDE.md` 三-1）。
- **评审 - 返工流程（有界）**：全新实例评审出 verdict → 非 PASS 把 mustFix 发给实现者（不同实例）返工 → **新的**评审实例重新独立核查（不信「已改」自述）→ 最多 2 轮返工；仍不过则如实输出未完成报告，不得声称完成（详见 `review-rubric.md` 评审-返工流程）。
- **评审者的工具与纪律**：`workflow-reviewer` 用 Read / Bash / Grep / Glob / WebFetch 实际核查，不修改任何交付物，证据不足即降级——本文第五节的「达标证据 vs 反模式」就是它逐维核查时的判据来源。

> 一句话：`workflow-reviewer` 是「谁来评」，本文是「按什么评、怎么排名」，`review-rubric.md` 是「评的时候手边那张卡」。
